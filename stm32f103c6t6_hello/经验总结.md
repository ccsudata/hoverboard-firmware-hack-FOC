# stm32f103c6t6_hello 多次修改后最终跑通 LED + UART 的经验总结

## 目标

把这次工程的所有问题按**层次**和**类别**整理成一份可复盘的总结，重点回答：

1. 这次一共遇到了哪些问题
2. 为什么没有一次成功
3. 最终哪些改动真正起作用
4. 以后如何避免再踩同类坑

---

## 一、最终可用的稳定方案

这次最终可用的方案是：

- 使用 `startup_stm32.s` 作为启动代码
- 使用 `stm32f103.ld` 指定正确的 Flash/RAM 布局
- `main.c` 只保留应用逻辑：时钟、GPIO、UART、SysTick、主循环
- 统一通过 `build.sh` 编译，避免每轮手工拼参数导致不一致
- 先验证 LED，再验证 UART，避免把两个问题混在一起分析

这套方案的核心是：**启动链路、链接脚本、运行时内存、外设配置、真实验证必须同步成立。**

---

## 二、按层次分类的问题总结

### 2.1 启动层问题

#### 问题 1：启动方式和成功工程不一致

最初 `stm32f103c6t6_hello` 和成功工程的启动方式不统一：

- 成功工程采用的是最小启动思路
- 当前工程采用的是启动文件 + C 代码的方式

这导致两边的镜像结构和控制流完全不同，后续排查时很难直接对照。

**后果：**

- 代码能编译，但不能快速判断问题到底出在启动流程还是应用逻辑
- 每次改动都像是在换一套工程结构

**经验：**

- 一个工程只保留一条启动路线
- 先定好基准方案，再开始做功能扩展

---

#### 问题 2：向量表定义方式混乱

早期曾出现两种向量表定义方式并存：

- `main.c` 中定义 `.vectors`
- `startup_stm32.s` 中定义 `.isr_vector`

这会让 Reset 入口和中断表源不唯一。

**后果：**

- Reset 入口不明确
- 向量表位置可能不符合预期
- CPU 可能跳到错误的位置，导致“看起来像没有运行”

**经验：**

- 只保留一种向量表定义方式
- 让启动文件、链接脚本、C 代码都用同一套约定

---

### 2.2 链接层问题

#### 问题 3：链接脚本和启动文件不匹配

`startup_stm32.s` 里使用的是 `.isr_vector`，而早期版本的 C 代码和链接脚本约定没有完全一致。

**后果：**

- 链接成功，但运行时行为异常
- 镜像里看起来有代码，但执行路径并不可靠

**经验：**

- 链接脚本、启动代码、向量表的 section 名称必须完全一致
- 不能靠“看起来能编译”来判断链接关系正确

---

#### 问题 4：RAM 大小配置错误（最致命的根因）

这一步是这次失败里最关键、最有证据的根因。执行 `monitor reset halt` 后，GDB 打印的关键寄存器是：

- `xPSR: 0x01000000`
- `pc: 0x080001b0`
- `msp: 0x20005000`

#### 为什么这说明失败了

STM32F103C6T6 的 SRAM 范围是：

- `0x20000000` 到 `0x20002800`

而 `0x20005000` 明显超出 C6T6 的物理内存范围。也就是说，**这不是栈太大，而是地址本身根本不存在。**

#### 这一步是如何导致 HardFault 的

- 复位时 CPU 只是把 `0x20005000` 读进了 `msp`
- 硬件不会立即报错
- 之后一旦执行到 `bl main`，Cortex-M3 会开始使用这个栈顶去压栈
- 访问 `0x20005000` 时，总线拒绝访问
- 触发 `HardFault`
- 由于当时没有实现 `HardFault_Handler`，最终掉进 `Default_Handler`

#### 根因定位

根因就在 `stm32f103.ld`：

- RAM 被配成了 **C8T6 的 20KB**
- 但实际芯片是 **C6T6 的 10KB**

#### 最终修复

最终版本把 RAM 改成了真实芯片规格：

- `RAM (rwx) : ORIGIN = 0x20000000, LENGTH = 10K`

这样 `_estack` 才会正确落在 `0x20002800`，主程序才能安全进入 `main()`。

---

### 2.3 运行时层问题

#### 问题 5：运行时变量被放进了错误的存储区

早期版本里，像 `systick_ms`、`last_toggle` 这类变量没有被正确落到 RAM。

**后果：**

- 代码能编译
- 运行时状态不可靠
- 变量行为与预期不一致

**经验：**

- 运行时数据必须明确落在 RAM
- `.data`、`.bss` 的初始化必须可验证

---

#### 问题 6：LED 翻转条件逻辑错误（逻辑雪崩）

这次真正卡住 LED 的真相非常纯粹：**核心是主循环里的逻辑雪崩**。

原来的写法里，`n` 在 `while(1)` 中持续执行 `n += 100`。
由于单片机循环速度极快，`n` 会在每一轮都无条件自增，几乎没有机会等到条件成立。

**结果：**

- 很快就会把 `n` 累加到非常大的数值
- `systick_ms` 只能按 1ms 增长
- `n` 的增长速度远远超过 `systick_ms`
- `if (systick_ms >= n)` 一直无法满足
- `PC13` 就永远不会翻转

这就是“**逻辑雪崩**”的本质：不是单纯的延时问题，而是**计数器本身被错误地做成了一个无法收敛的绝对增长变量**。

**修复方式：**

- 改成固定周期判断，而不是让 `n` 失控增长
- 让 LED 翻转条件基于 `systick_ms` 的相对时间差来判断

**结果：**

- 修复后，LED 真正恢复闪烁

---

#### 问题 7：SysTick 时钟源配置瑕疵（次要原因）

这一步是“为什么 LED 还很慢、为什么计时不是预期值”的次要根因。

原来你写的 `SYSTICK_CTRL = 0x03`，只开了：

- Bit 0：`ENABLE`
- Bit 1：`TICKINT`

但**没有设置 Bit 2：`CLKSOURCE`**。

在 Cortex-M3 上，缺少 `CLKSOURCE` 时，SysTick 默认使用的是**外部低速时钟**，而不是处理器系统时钟。

**结果：**

- 你本来给 `SYSTICK_LOAD = 71999`，这是按 72MHz 设计的
- 但实际在低速时钟下，定时周期被拉长成约 8ms
- `systick_ms` 增长速度被明显拖慢
- 这让“逻辑雪崩”问题看起来更像是“完全没反应”

**经验：**

- SysTick 的时钟源一定要和实际系统时钟对应
- 不能只看重装载值，还要看 `CLKSOURCE` 是否被正确置位

---

#### 问题 8：时钟未真正切换到 PLL（早期版本的关键证据）

你还记得调试时看到的证据：

- `RCC_CFGR = 0x0`
- `RCC_CR = 0x4d83`

这说明早期版本里：

- HSI 已经开启
- HSE 没有开启
- PLL 也没有开启
- 系统时钟仍然停留在默认的 8MHz HSI

这就解释了为什么你之前配置的 `SYSTICK_LOAD` 和串口波特率参数会出现“看起来像对，实际却慢/不准”的情况。

**最终正确做法：**

- 继续使用内部 HSI
- 启动 PLL
- 把系统时钟拉到 **64MHz**
- 同步把 `SYSTICK_LOAD` 改成 `63999`

这样才能和当前代码里的实际时钟配置一致。

---

### 2.4 外设层问题

#### 问题 7：LED 和 UART 被混为一类问题

在一开始，很多问题被当作“整个系统没跑起来”来处理，实际上 LED 和 UART 是两个不同层次的问题。

**实际情况：**

- 修复 LED 后，LED 可以正常闪
- UART 仍然没有输出

这说明：

- LED 问题和 UART 问题不是同一个根因
- 之前的排查把两个问题混在一起了

**经验：**

- 先把 LED 跑通，再单独验证 UART
- 外设问题应该分开拆解，不要混成一个大问题

---

#### 问题 8：UART 只是“代码有函数”，并不代表实际输出成功

一度只看到 `uart_init()`、`uart_send_string()` 这些函数存在，就误以为串口链路已经没问题。

**问题点：**

- 没有验证 TX 是否真的到达外部串口工具
- 没有验证引脚、波特率、USART 使能是否真实生效
- 没有把“代码存在”与“物理输出可见”区分开

**经验：**

- UART 必须单独做真实验证
- 需要确认：
  1. TX 引脚是否正确
  2. 波特率是否匹配
  3. USART 使能位是否正确
  4. 外部接收端是否真正收到数据

---

### 2.5 验证层问题

#### 问题 9：只验证了编译，没有完整验证运行

早期很多轮次只停留在“能编译、能链接”这一层。这个结论并不等于“板子真的跑起来了”。

**后果：**

- 代码看起来正确
- 但硬件上没有实际行为

**经验：**

- 编译成功只是第一步
- 还必须验证：
  1. 复位后是否进入 `main`
  2. `msp` 是否合法
  3. `pc` 是否落在正确位置
  4. LED 是否真实闪烁
  5. UART 是否真实输出

---

#### 问题 10：没有建立稳定的验证闭环

缺乏“每一轮都验证一个明确结论”的习惯，导致问题反复被覆盖。

**经验：**

- 每次只验证一个核心问题
- 每次记录“已经解决什么、还剩什么”
- 不能把多个猜测混在一起做一次性修复

---

## 三、为什么这次没有一次成功

这次没有一次成功，不是因为只有一个 bug，而是因为**多个层次的问题叠加在一起**。

### 3.1 关键原因汇总

1. **启动流程不统一**，导致问题边界不清楚
2. **向量表定义方式冲突**，使启动入口不稳定
3. **链接脚本和启动文件约定不一致**
4. **RAM 大小写错**，导致 `msp` 指向非法地址
5. **运行时变量放区不可靠**
6. **LED 时间逻辑错误**，导致看起来像功能没跑
7. **LED 与 UART 被混在一起分析**，掩盖了真实因果
8. **只看代码，不验证硬件输出**
9. **缺乏可复用的验证闭环**

### 3.2 为什么会“每次都只修掉一部分”

因为问题不是单点，而是多层次叠加：

- 层次上有启动、链接、运行时、外设、验证问题
- 结果上有“代码能编译”但“硬件不跑”的现象
- 所以每次往往只解决一部分，另外一部分继续掩盖真实问题

这也是为什么“没有一次成功”的根本原因。

## 四、最终成功的关键经验

### 4.1 最关键的规则

1. **先固定启动方案**，不要来回换模板
2. **只保留一种向量表定义方式**
3. **链接脚本必须和芯片型号匹配**
4. **栈顶地址必须合法**
5. **运行时数据必须落在 RAM**
6. **先验证 LED，再验证 UART**
7. **每一轮只验证一个明确目标**
8. **必须做真实硬件验证，而不是只看代码**

### 4.2 最终可复用的工程顺序

建议以后按下面顺序做：

1. 先固定 `startup_stm32.s` 和 `stm32f103.ld`
2. 先做最小 LED 工程
3. 确认 `msp`、`pc`、Reset 流程正确
4. 再接入 SysTick
5. 再做 UART
6. 最后做真实串口输出验证

## 五、未来可直接复用的经验清单

### 5.1 文件层

- `build.sh`：统一编译入口
- `startup_stm32.s`：唯一的启动代码
- `stm32f103.ld`：必须与芯片型号匹配

### 5.2 调试层

- 先看 `monitor reset halt` 的 `pc` 和 `msp`
- 先确认栈顶是否落在芯片真实 RAM 范围内
- 先确认 `HardFault` 是否被触发
- 先确认 `main()` 是否真的被执行

### 5.3 外设层

- LED：先验证 GPIO 是否真正翻转
- UART：单独验证串口波特率、引脚、使能位和外部接收端

## 六、这次工程的最终结论

这次最终成功的真正原因不是某一个单独改动，而是下面几件事一起收敛到了正确状态：

- 启动方案统一
- 向量表唯一
- 链接脚本改成了 C6T6 的真实 RAM 配置
- 运行时变量落在 RAM
- LED 时间逻辑修正
- UART 被单独拆开验证
- 每一步都做了真实验证，而不是只凭代码判断

### 最值得记住的一句话

> **裸机开发不是“写代码就能跑”，而是“启动链路 + 链接脚本 + 内存布局 + 外设配置 + 真实硬件验证”五层同时成立。**

如果以后再做类似工程，建议把这份总结当成模板，严格按“启动层 → 链接层 → 运行时层 → 外设层 → 验证层”的顺序去排查。
